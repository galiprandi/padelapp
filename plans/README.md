## Findings considered and rejected

- **View Transitions (cross-document)**: low priority for MVP. The skeletons
  and non-blocking layout address the perceived performance problem more
  directly. View Transitions can be added later as a progressive enhancement.
- **`force-dynamic` on `/match`**: valid finding (confirmed matches rarely
  change but every visit hits the DB), but deferred to a second batch. The
  page needs to be split into a cached confirmed-matches list and a dynamic
  pending-actions section — that's a medium effort refactor, not a quick win.
- **Over-fetching (`user: true` in match-queries.ts)**: valid finding (full
  User record per player per match), but deferred. Fixing it requires
  changing the `MatchResultCompactMatch` type and all consumers — medium
  effort, higher risk of breaking the match card rendering.
- **Waterfall queries in ranking page**: valid finding (`auth()` →
  `searchParams` → `findMany` → `findUnique` sequential), but `auth()` and
  `searchParams` are both fast (no DB). The real win is parallelizing the two
  `prisma.user` queries, which is a small change but deferred to avoid
  overlapping with the loading.tsx work.
