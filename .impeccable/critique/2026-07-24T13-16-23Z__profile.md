# Design Critique: Padel Red Profile Edit Page (/me/profile)

## Design Specificity Verdict

**Category-interchangeable with mild padel flavoring.** The only authentically padel-specific elements are the avatar preset names ("Pala", "Smash", "Volea", "Globo", "Efecto") and the phrase "Alias en la cancha." Remove those and this is a generic SaaS profile edit page. The copy "Personalizá tu identidad en la red" is vague corporate-speak that doesn't ground the page in padel culture or the club context. The page fails to leverage the product's unique positioning (implicit padel network, turn salvation) in its design or messaging.

## Overall Impression

The page is functional and clean, but structurally confused. It shows the avatar twice, mixes 7 avatar options in one row, and buries a security feature (passkeys) without context. The single biggest opportunity: collapse the redundant avatar display and separate the avatar sources into clear stacked sections — this alone would halve the cognitive load.

## What's Working

1. **Padel-specific avatar preset names** — "Pala", "Smash", "Volea", "Globo", "Efecto" ground the page in padel culture. These are the only elements that truly signal "this is a padel app."
2. **Real-time avatar preview with instant feedback** — the preview updates immediately when selecting avatars, reducing uncertainty. The X button to remove is well-placed.
3. **Clear alias validation with Spanish error messaging** — the 2-30 character validation with "Usá entre 2 y 30 caracteres" is helpful and localized. The helper text "Este nombre verán tus rivales en partidos y ranking" explains the value well.

## Priority Issues

### [P0] Redundant avatar display creates hierarchy confusion
**What**: Avatar is displayed twice — once in a static card at the top (page.tsx lines 47-78) and again as an interactive preview in the form (profile-form.tsx lines 97-120). Name/alias also shown twice.
**Why it matters**: Users don't know which is the "real" avatar. The top card is static; the form preview is interactive. This breaks "what you see is what you get" and wastes vertical space on mobile.
**Fix**: Remove the top avatar card entirely. The form preview is sufficient and updates in real-time. If a public-facing preview is needed, make it update with form changes and label it "Vista previa."
**Suggested command**: `$impeccable distill /me/profile`

### [P0] Avatar selector mixes 3 sources in one row (7 options > 4)
**What**: Google avatar, 5 presets, and custom URL input are all in one horizontal row — 7 visible options exceeding the cognitive load threshold.
**Why it matters**: Cognitive load increases significantly with >4 options. The three sources are not visually separated, making it hard to scan. At the club, on mobile, with hands occupied, this is too much noise.
**Fix**: Separate into stacked sections with headers: "Tu foto de Google" (if available), "Avatares deportivos" (presets), "Imagen personalizada" (custom URL). Vertical stacking with gap-3.
**Suggested command**: `$impeccable shape "avatar selector"` then implement, or `$impeccable layout /me/profile`

### [P1] `text-[8px]` violates DESIGN.md §1.8 (micro-typography ban)
**What**: The Google avatar "G" badge uses `text-[8px]` (profile-form.tsx line 148). DESIGN.md §1.8 bans typography below 11px. The detector missed this; manual cross-reference caught it.
**Why it matters**: Direct violation of the design system. Illegible at 8px on mobile. Also uses `bg-background/90` (semi-transparent), borderline per §1.3.
**Fix**: Remove the "G" badge entirely — the Google avatar is already visually distinct. If a label is needed, use `text-xs` but the badge adds noise without value.
**Suggested command**: `$impeccable polish /me/profile`

### [P1] Cancel button lacks confirmation for unsaved changes
**What**: Cancel is a full-width ghost button linking to /me (profile-form.tsx lines 230-232) with no confirmation if there are unsaved changes.
**Why it matters**: Users can accidentally lose changes. At the club, with hands occupied, this is a high-friction mistake. Violates "cero fricción."
**Fix**: Add dirty state tracking (compare current alias/image with initial values). Show a confirmation dialog ("¿Descartar cambios sin guardar?") when dirty. Use shadcn AlertDialog.
**Suggested command**: `$impeccable harden /me/profile`

### [P2] "Personalizá tu identidad en la red" is vague/corporate
**What**: Page description says "Personalizá tu identidad en la red" (page.tsx line 18).
**Why it matters**: Doesn't speak to the user at the club. "La red" is vague. Sounds like corporate SaaS, not a padel app. Fails the product principle of "voz cercana, deportiva, amateur."
**Fix**: Use concrete padel language: "Cómo te ven los demás jugadores en el ranking y los partidos."
**Suggested command**: `$impeccable clarify /me/profile`

### [P2] PasskeyManager appears without section separation
**What**: PasskeyManager is rendered below ProfileForm with no section header or visual separation (page.tsx line 86).
**Why it matters**: Users won't understand that passkeys are a separate concern from profile identity. No label explains what passkeys are. First-timers will be confused.
**Fix**: Add a section header "Seguridad" with helper text "Iniciá sesión sin contraseña usando tu huella o face ID." Use gap-6 separation.
**Suggested command**: `$impeccable layout /me/profile`

### [P3] `active:scale-[0.98]` on avatar toggle buttons (DESIGN.md §1.5 vs §7.1 ambiguity)
**What**: Avatar selector buttons use `active:scale-[0.98]` (profile-form.tsx lines 132, 161). DESIGN.md §1.5 bans tactile scale effects; §7.1 allows it for "primary and secondary buttons" only.
**Why it matters**: These are small toggle buttons, not primary/secondary action buttons — likely fall outside the §7.1 carve-out. Policy contradiction needs resolution.
**Fix**: Remove `active:scale-[0.98]` from avatar toggle buttons, or resolve the §1.5 vs §7.1 contradiction in DESIGN.md by clarifying scope.
**Suggested command**: `$impeccable polish /me/profile`

## Persona Red Flags

### First-Timer (joined via WhatsApp link, setting up profile)
- "Personalizá tu identidad en la red" doesn't explain what this page is for or why it matters.
- No guidance on what makes a good alias — first-timers may not understand this is their "padel name."
- PasskeyManager appears with no explanation — first-timers won't know what passkeys are.
- Empty state: when alias is empty and no image selected, there's no guidance or "skip for now" option.

### Mobile-on-the-go (at the club, coordinating via WhatsApp)
- Avatar selector has 7 options in one row — too much to scan while distracted.
- Custom URL input is small (h-10) and hard to tap accurately with one hand.
- Cancel button is full-width and easy to accidentally tap.
- The redundant avatar display wastes vertical space, pushing content below the fold.

### Power User (organizer who plays frequently)
- No bulk editing or quick actions.
- Can't quickly cycle through avatars with swipe/gesture.
- No keyboard shortcuts.
- Explicit save/cancel is unnecessary friction for a simple identity update.

## Minor Observations

- `loading.tsx` skeleton uses `rounded-full` for the avatar (line 13) but the actual page uses `rounded-xl` — inconsistency between loading and loaded states.
- ShareButton in the top card is a nice touch but not clear what it shares without tapping.
- Avatar preset URLs use DiceBear API — external dependency with no fallback.
- The Google avatar badge uses `bg-background/90` (semi-transparent) — borderline per DESIGN.md §1.3.
- No empty state guidance when user has no alias and no image — the form feels incomplete.

## Questions to Consider

1. Why does the page show the avatar twice? Is the top card meant to be a "public profile preview" or just redundant display? If it's a preview, why doesn't it update in real-time as the user edits the form?
2. The product says "cero fricción es obligatoria" and the user is "en el club, con las manos ocupadas." Why does this page require an explicit save action instead of auto-saving with an "undo" toast?
3. Should passkey management be moved to a separate `/me/security` route? What's the user journey that justifies mixing identity editing with security settings on the same screen?
